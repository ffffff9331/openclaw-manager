# openclaw-manager v2 阶段二人工回归速测版

日期：2026-04-04
用途：给老大或后续会话一个**最短可执行的点按顺序**。不代替 `stage2-manual-test-run.md`，而是它的速查版。

---

## 一、先记住总原则
- 只测 **当前选中实例**
- 不把“已完成开发”当成“已正式通过”
- 远端实例只测**引导 / 管理 / 展示**，不测“远端自动安装”
- 若某项失败，记录页面、动作、报错，不当场扩新功能

---

## 二、开测前 30 秒检查
- [ ] `npm run build` 已通过
- [ ] 已确定当前实例（`local` 或 `remote`）
- [ ] 若测安装：准备一个 `local` 实例
- [ ] 若测 Docker / NAS 引导：准备一个 `remote` 实例
- [ ] 若测备份：准备一个可写目录

---

## 三、真实 Tauri 手测最短路径（建议先跑这一版）
1. 先用 `local` 实例连续点：`Settings → Models → Skills → Gateway → Doctor → Tasks → AppLogs`
2. 每页只验证一件最核心动作是否能触发、是否有结果区/状态区回显
3. 只要出现异常，立即记录：页面名 / 点击动作 / 结果文案，不在手测现场扩功能
4. `remote` 实例只补测 `Settings` 的 Docker / NAS 引导与复制模板，不测远端自动安装
5. 测完后直接回填“通过 / 不通过 / 异常一句话”，再决定是否返修

### 本轮建议直接照抄的最短执行版
- `SettingsPage`：点一次安装/部署主入口，再点一次备份动作，确认有结果区或动作反馈
- `ModelsPage`：点一次“刷新模型”或连通性测试，再点一次“一键添加预设”，确认有状态反馈
- `SkillsPage`：点一次“刷新技能”，再点一次启用/禁用或打开目录，确认有反馈
- `GatewayPage`：点一次“刷新”，再点一次“启动/停止/重启”中的任意一个，确认状态区 / 日志区 / 最近动作区会变化
- `DoctorPage`：点一次“一键诊断”，确认结果区从待检测进入执行或结果态
- `TasksPage`：点一次“刷新任务”，再点一次“新建任务”或“立即运行”，确认有状态反馈
- `AppLogsPage`：点一次“刷新日志”，确认空态 / 非空态 / 错误提示可读

### 每页只记 3 件事
1. 动作有没有发出去
2. 页面有没有结果区 / 状态区回显
3. 提示文案是不是动作化、可读、没越界

## 四、最小执行顺序（建议 7 步）

### Step 1：先看当前实例有没有串线
页面：任意页切到 `Settings / Models / Skills / Gateway / Doctor / Tasks / AppLogs`

过线标准：
- 所有页面顶部实例名称 / 类型 / 地址一致
- 切换实例后，各页都跟着变

如果失败，记录：
- 哪个页面没跟着切
- 旧实例 / 新实例分别是谁

---

### Step 2：测 `SettingsPage`
先测安装 / 部署，再测备份。

#### 2A. 安装 / 部署
`local` 重点看：
- 能否检查安装状态
- 能否看到本地安装入口

`remote` 重点看：
- 是否明确显示 Docker / NAS 引导
- 是否能看到：
  - `docker-compose.yml`
  - `docker run`
  - NAS 检查清单
- 复制按钮是否可用
- 没有假装“远端一键安装”

#### 2B. 备份
重点点：
- 预览备份
- 创建备份
- 校验备份
- 还原备份入口

过线标准：
- 页面能输入参数
- 按钮能触发
- 结果区有输出
- 失败时是动作化提示，不是一坨裸异常

---

### Step 3：测 `ModelsPage`
重点点：
- 一键添加预设
- 新增自定义模型
- 编辑模型
- 设为默认
- 删除自定义模型
- 单项连通性测试
- 批量连通性测试
- 上移 / 下移

过线标准：
- 操作后有状态反馈
- 连通性结果能显示在页面
- 顺序调整后刷新结果正确
- 页面文案明确是“按需重载 Gateway”，不是自动重启 runtime

---

### Step 4：测 `SkillsPage`
重点点：
- 刷新技能列表
- 启用技能
- 禁用技能
- 本地下打开目录入口

过线标准：
- 列表能加载
- 启用 / 禁用后有状态回显
- `remote` 下若尝试打开目录，得到明确边界提示

---

### Step 5：测 `GatewayPage`
重点点：
- 打开页时自动刷新状态
- 启动 / 停止 / 重启 / 刷新
- 看 LaunchAgent 状态 / 日志 / 历史 / duration

过线标准：
- 控制动作后，状态与日志会刷新
- 页面语义是“已投递 / 已发起”，不是假装同步完成

---

### Step 6：测 `DoctorPage` + `AppLogsPage`
#### Doctor
- 一键诊断
- 刷新诊断结果

#### AppLogs
- 刷新日志
- 看空态 / 非空态是否合理

过线标准：
- 都围绕当前实例
- 结果能看到
- 失败提示可读

---

### Step 7：测 `TasksPage`
重点点：
- 看 cron 状态
- 创建任务
- 编辑任务
- 启用 / 禁用
- 立即运行
- 删除
- 看运行记录

过线标准：
- 维护闭环打通
- 页面未宣称支持高级字段
- 不同动作的失败提示可区分

---

## 四、速测结论模板
测完后只要填这几行：

- 当前实例串线：通过 / 不通过
- Settings：通过 / 不通过
- Models：通过 / 不通过
- Skills：通过 / 不通过
- Gateway：通过 / 不通过
- Doctor：通过 / 不通过
- Tasks：通过 / 不通过
- AppLogs：通过 / 不通过

### 最终判断
- [ ] 阶段二当前批次具备人工通过条件
- [ ] 需要先修回归问题再继续

### 若不通过，优先记三件事
1. 哪一页
2. 点了什么动作
3. 实际报错 / 异常现象

---

## 五、汇报时怎么说
### 可以说
- 阶段二首批核心管理能力开发已基本成形
- 当前正在做人工回归
- 某页通过 / 某页有回归问题

### 不要说
- 阶段二已正式完成
- 已支持远端自动安装
- 已完成模型拖拽排序
- manager 已接管 openclaw 本体执行逻辑
