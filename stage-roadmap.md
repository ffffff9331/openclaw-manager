# openclaw-manager v2 阶段划分与验收标准

日期：2026-04-03
状态：生效中
用途：作为 `openclaw-manager` 后续推进、汇报、回归和验收时的统一阶段口径。后续阶段描述以本文件为准，不再口头漂移。

---

## 阶段 1：底座重构（基础架构，最重要）

### 范围
- 前端项目结构模块化重构（`src/pages/`、`src/components/`、`src/lib/`、`src/stores/` 等）
- 实例模型（`Instance`）完整定义与 TypeScript 类型
- 服务层抽象（`instanceService.ts`、`apiClient.ts` 等，所有功能基于当前选中实例）
- 状态管理引入（推荐 Zustand）
- 开发 / 构建脚本规范化（`dev.sh`、`build.sh`、`release.sh` 等）
- Tauri 配置更新（局域网访问权限、安全配置等）
- 数据持久化机制（实例列表、当前实例、设置等）

### 验收标准
- 项目结构清晰
- 能创建、保存、切换实例
- 底层框架稳定

---

## 阶段 2：核心管理能力

### 范围
- 一键安装 OpenClaw（支持本地 + Docker / NAS 部署引导）
- 配置备份 / 还原（支持 zip 导出 + volume 提示）
- 技能管理（列表、启用 / 禁用、简单编辑）
- 定时任务管理（查看、创建、编辑、删除）
- 模型管理（批量连通性测试、排序调整 / 拖拽交互、一键添加按钮）
- 所有核心功能适配“当前选中实例”

### 验收标准
- 用户选中实例后，能正常使用安装、备份、模型等主要运维功能

---

## 阶段 3：Docker / NAS / Web 化支撑

### 范围
- Web / API 适配层（统一 HTTP 调用方式，为后续 Web 版打基础）
- 推荐 `docker-compose.yml` 模板（包含 OpenClaw + LiteLLM，带详细注释）
- NAS 部署文档与引导（群晖 Container Manager、威联通 Container Station）
- 卷挂载、权限处理、跨域配置、host 网络模式等实用方案
- 实例类型区分（`local` / `docker` / `nas` / `remote`）
- 常见问题处理与一键复制命令

### 验收标准
- Docker 和 NAS 用户能按照文档顺利部署并通过 manager 连接管理

---

## 阶段 4：实例接入增强

### 范围
- 本机实例自动检测
- 已保存实例管理与切换
- 手动添加实例（地址、端口、Token、名称等）
- mDNS / Bonjour 自动发现（默认关闭，设置中可开启）
- 保守特征探测（常见端口 + 特征接口验证，如 `/health` 或 gateway status）
- 发现结果展示为卡片，支持一键保存为实例
- 安全开关与提示文案（mDNS 风险提示）
- 实例在线状态刷新机制

### 验收标准
- 用户能在局域网内方便发现并接入 OpenClaw 实例，自动发现机制稳定且安全

---

## 当前阶段判断
- 阶段 1：底座重构已基本完成，当前更接近“代码与文档收口完成；正式完成待人工实测盖章”
- 阶段 2：首批核心管理能力开发已基本成形；web preview 管理面回归证据已基本补齐，当前重点不再是继续补 preview，而是转入真实 Tauri 宿主环境人工回归
- 阶段 2 当前已落地的代表项包括：
  - `SettingsPage`：本地安装检查 / 本地安装入口 / 远端 Docker-NAS 引导模板 / 备份创建-校验-还原入口
  - `ModelsPage`：一键添加预设 / 单项批量连通性测试 / 可持久化顺序调整
  - `SkillsPage`：技能列表 / 启用禁用 / 打开目录入口
  - `GatewayPage`：状态读取 / 动作投递 / LaunchAgent 结构化状态与历史
  - `TasksPage`：cron 第一版维护闭环
- 阶段 2 当前已完成的 preview 管理面补证范围包括：首页初始化、`SettingsPage`、`ModelsPage`、`SkillsPage`、`GatewayPage`、`DoctorPage`、`AppLogsPage`、`TasksPage`
- 阶段 3：当前批次已完成主要代码收口：服务语义统一、实例类型区分、Docker/NAS 部署引导、Web/API 适配层已落地并多轮 build 通过；正式完成仍待与后续阶段一起做人工验收补章
- 阶段 4：代码侧已基本收口，但尚未正式完成：
  - 已完成：本机实例检测入口、手动添加增强、局域网发现入口（mDNS 开关 / 扫描按钮 / 保守探测结果 / 保存为实例）、在线状态刷新最小闭环（实例状态探测、侧栏手动刷新、轻量轮询、状态展示）
  - 未完成：阶段 4 的人工验收补章与最终正式完成结论

---

## 推进规则
- 严格按阶段推进：完成一个阶段，验收一个阶段，验收通过后再进入下一阶段。
- 如果某阶段内部已经提前做出一部分后续能力，也要先收成明确批次，再单独验收，不继续无限外扩。
- 当前项目的执行顺序应理解为：
  1. 阶段 1 正式验收盖章
  2. 阶段 2 首批能力人工回归与结论
  3. 阶段 2 后续批次 / 体验增强（如拖拽交互）
  4. 阶段 3 / 阶段 4
- 当前未通过验收的内容，不能在汇报口径里写成“已正式完成”。
