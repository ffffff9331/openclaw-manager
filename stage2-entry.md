# openclaw-manager v2 阶段二入口判断

日期：2026-04-04
状态：阶段二首批能力已基本成形（代码收口完成；人工回归待盖章）

## 一、当前总判断
阶段二当前已经不再是“刚起步”或“只有骨架”，而是：

> **首批核心管理能力开发已基本成形；web preview 管理面回归证据已基本补齐，当前重点已转为真实 Tauri 宿主环境人工回归。**

这意味着后续推进要带着两个前提：
1. 不把“已完成开发”误写成“已正式验收完成”
2. 不再回头把已经落地的真实闭环反复说成“还只是骨架”

---

## 二、哪些问题已经定性，不再反复回抠

### 1. `allowLanAccess`
当前已明确属于**真闭环能力**：
- 设置项已持久化
- 设置页可见可改
- `AddInstanceModal -> validateInstanceBaseUrl(...)` 已真实接入新增远端实例地址校验链

因此后续不应再把它当作“只有设置、没有行为”的半接线功能。

### 2. `mdnsEnabled`
当前已明确属于**预留偏好 / 规划中**：
- 仍保留数据位与界面说明
- 尚未接入自动发现流程
- 不能视为当前版本真实可用能力
- 不应再出现在阶段二通过证据里

因此阶段四若继续推进它，应视为新增真实能力，而不是回头补阶段一/二遗留。

### 3. 阶段二当前首批能力
当前已经形成真实代码闭环的代表项包括：
- `SettingsPage`：本地安装检查 / 本地安装入口 / 远端 Docker-NAS 引导模板 / 备份创建-校验-还原入口
- `ModelsPage`：一键添加预设 / 单项批量连通性测试 / 可持久化顺序调整
- `SkillsPage`：技能列表 / 启用禁用 / 打开目录入口
- `GatewayPage`：状态读取 / 动作投递 / LaunchAgent 结构化状态与历史
- `TasksPage`：cron 第一版维护闭环

因此后续不应再把阶段二描述成“只有安装 / 备份骨架”。

---

## 三、当前更重要的不是继续扩，而是收口
当前更合理的下一步已经不是继续扩张新页面，而是：
- 更新人工回归清单
- 明确边界口径
- 区分“已完成开发”与“待人工盖章”
- 收掉明显的交互打磨项是否值得继续做

这也是为什么当前阶段二工作重点应从“继续加能力”切换到“形成可验收结论”。

---

## 四、模型管理这一块的准确口径
阶段二文档最初写的是：
- 批量连通性测试
- 拖拽排序
- 一键添加按钮

当前真实状态应写成：
- 一键添加预设：已落地
- 单项 / 批量连通性测试：已落地
- 排序调整能力：已落地（当前为上移 / 下移的可持久化入口）
- 拖拽交互：仍可视为后续体验增强

也就是说：
> **排序能力已具备，但拖拽交互尚未单独打磨。**

---

## 五、安装能力这一块的准确口径
当前真实状态不是“远端自动安装已完成”，而是：
- `local`：支持安装检查与本地安装入口
- `remote`：支持 Docker / NAS 引导、模板展示、复制入口

因此更准确的说法是：
> **安装能力当前收口为“本地安装 + 远端部署引导”，而不是远端自动执行。**

这条不是缺陷，而是为了守住 manager 与 openclaw 本体边界后的主动收口。

---

## 六、阶段二当前建议起手顺序（更新后）
1. **优先：转入真实 Tauri 宿主环境人工回归**
   - 先按 `stage2-manual-test-quick.md` 跑最短执行版
   - 再按 `stage2-manual-test-run.md` 逐页回填真实结果
   - 时间有限时优先：`SettingsPage` → `ModelsPage` → `GatewayPage` → `TasksPage`
2. **其次：保持入口 / 结论 / 回归文档口径一致**
   - `stage-roadmap.md`、`stage2-entry.md`、`stage2-conclusion.md`、`stage2-manual-test-quick.md`、`stage2-manual-test-run.md` 不再回到旧口径
3. **最后：再评估是否继续做体验增强**
   - 例如模型拖拽交互
   - 这应作为后续批次，而不是继续混在首批能力验收口径里

---

## 七、阶段二推进时的约束
- 不把阶段一状态写成“已正式完成”
- 不把阶段二当前已落地能力回退描述成“只有骨架”
- 不为了图快把 openclaw 本体能力揉进 manager
- 继续保持每刀后立即 `npm run build`
- 当前优先级应是“能否进入人工验收”，不是“还能不能再多做一页”

---

当前阶段状态应统一理解为：

- **阶段一代码与文档收口：已完成**
- **阶段一正式完成：待人工实测盖章**
- **阶段二首批能力开发：已基本成形**
- **阶段二正式通过：待人工回归结论**

也就是说，当前最合理的下一步是：

> **围绕 `SettingsPage` / `ModelsPage` / `SkillsPage` / `GatewayPage` / `TasksPage` 完成阶段二人工回归结论，而不是继续无限外扩功能面。**
